Namespaces in Okteto offer a flexible way to organize your Development Environments. A Namespace can last for a few weeks for developing a feature, or a couple days for a bug fix.
Not only do Namespaces give you an individual separation of context while you are working, but you can also use shared Namespaces to collaborate with you team before anyone pushes to git.

In Okteto, there are two types of Namespaces:

- ### Personal Namespaces:
  Automatically generated for each user, this Namespace is uniquely named after the user's username or email, ensuring a personalized environment. You can't delete these Namespaces, but you can destroy their resources to reuse the Namespace.

- ### Non-Personal Namespaces:
  These are additional Namespaces created by users or through automation, facilitating broader project collaboration and management. Users may delete them or transfer them to other users.

The Okteto Dashboard enables you to independently create, launch, and share Namespaces for yourself and your team.

## Manage your Namespaces

To create a Namespace, go to the Okteto dashboard, select your Namespace in the left sidebar, and at the bottom of the Namespace list select (+) New Namespace.

<p align="center">
  <Image
    src={require("@site/static/img/new-namespace+1.32.png").default}
    alt="new namespace dialog"
    width="800"
  />
</p>

:::tip
As a common pattern, you might name your Namespace by feature number, ticket number, or something similar.
Development Environments in Namespaces pull from git branches, which is why you may want to be running multiple Namespaces at the same time.
:::

The Okteto Admin Dashboard allows you to [sleep](admin/cleanup.mdx#manually-sleeping-resources) and [wake up](admin/cleanup.mdx#manually-wake-sleeping-resources) your Namespaces. Sleeping a Namespace shows a confirmation dialog; waking a Namespace takes effect immediately.

If you prefer to use the command line, here are the Namespace management options available:

- Create a Namespace with `okteto namespace create`
- Set your kubectl configuration to use your Namespace with `okteto kubeconfig`
- Delete a non-personal Namespace using `okteto namespace delete`
- [Sleep](admin/cleanup.mdx#manually-sleeping-resources) or [wake up](admin/cleanup.mdx#manually-wake-sleeping-resources) your Namespace with `okteto namespace sleep [name]` and `okteto namespace wake [name]` commands respectively

:::tip
Administrators can specify a global, default threshold for automatically sleeping a Namespace in the [garbage collection](self-hosted/helm-configuration.mdx#gc) configuration.
:::

## Sharing and Collaboration with Namespaces

By default, you are the only person with access to your Namespaces. Invite your teammates to enable true real-time collaboration.
Each user can collaborate on an application within a Development Environment before they push any code. Test better together.

To share a Namespace, go to the Okteto dashboard, select the Namespace you want to share, and select **Share** from the Namespace overflow menu (the three-dot menu).

  <Image
    src={require("@site/static/img/share+1.32.png").default}
    alt="share a Namespace"
    width="850"
  />

In the `Share` dialog, type the email address of the team members you want to share this Namespace with. Once you press save, Okteto updates the necessary permissions and notifies your teammates via email.


## Viewing Resource Metrics and Recommendations

### Resource Metrics View

With the addition of Resource Manager, there is a new “Metrics” tab in the Okteto Dashboard Namespace view. To access it, navigate to a Namespace, select a Development Environment’s Resource in the sidebar, and click on the Metrics tab under the endpoints section. This view provides detailed insights into the resource usage and performance of your Kubernetes workloads.

The Metrics tab is available for core resources, including Deployments, StatefulSets, Jobs, CronJobs, and Pods.

If the Resource Manager isn't enabled, the Metrics tab displays basic Storage and Pod metrics, with a CLI guide to view metrics in real-time.

## Namespace Administration

The Okteto platform takes care of all the required [RBAC roles and bindings](core/user-roles-and-permissions.mdx), resources [quotas](self-hosted/helm-configuration.mdx#quotas) and limits, and [network policies](self-hosted/helm-configuration.mdx#networkpolicies) so everyone in your team can safely deploy and develop applications in Kubernetes.

Through the Okteto Dashboard, Admins can perform all operations on any Namespace within the Okteto platform. This includes:
- Deleting, sharing, waking, sleeping, and marking Namespaces as persistent
- Deploying or deleting applications using both Okteto and `kubectl`
- Accessing all resources and sub-resources within any Development Environment
- [Transferring Namespace ownership](core/namespaces.mdx#transfer-namespace-ownership) between developers
- Admins can fine tune a Developer access to Namespaces by configuring [the `serviceAccounts` section within the Okteto Helm configuration](self-hosted/helm-configuration.mdx#serviceaccounts).
- Configure cpu, memory, pods and storage quotas for each Namespace using [the `quotas` section within the Okteto Helm configuration](self-hosted/helm-configuration.mdx#quotas).

### Transfer Namespace Ownership

Okteto administrators can transfer non-personal Namespaces between users. This process uses the Okteto UI and can be helpful when you need to manually remove resources from an inactive user. Personal Namespaces that belong to a user cannot be transferred.

To transfer a Namespace to a new owner, go to the **Okteto Dashboard -> Admin -> Namespaces** under Cluster Management, then locate the Namespace that you would like to transfer. Click on the three dots `...` to display the Namespace menu. From the Namespace menu select `Transfer Ownership`.

<p align="center">
  <Image
    src={require("@site/static/img/namespace-transfer-owner+1.42.png").default}
    alt="transfer a Namespace"
    width="650"
  />
</p>

When the Transfer Ownership modal appears, type the email address of the team member to whom you'd like to transfer the Namespace. Once you press confirm, ownership of that Namespace is immediately transferred.
If you wish, you can keep the current owner as a member of the Namespace by selecting the option in the Transfer Ownership modal. If this isn't selected the current owner immediately loses access.

<p align="center">
  <Image
    src={require("@site/static/img/namespace-transfer-modal.png").default}
    alt="Select a new Namespace owner"
    width="700"
  />
</p>

:::note
User specific variables don't transfer with a Namespace. If the Namespace being transferred has user specific variables, future build or destroy actions may fail.

You can add new user specific variables to a transferred Namespace to avoid future failures. Details on [adding variables are available here](core/okteto-variables.mdx).
:::

### Mark a Namespace as Persistent to Prevent it from Sleeping and Deletion

Administrators can mark a Namespace as `persistent` using the `Persistent` option to prevent it from sleeping, prevent it from being deleted, and exempt it from the [garbage collection](self-hosted/helm-configuration.mdx#gc) process.

To do so, navigate to the **Okteto Dashboard -> Admin -> Namespaces** under the Cluster Management section. Then locate the Namespace that you would like to mark as persistent. Click on the three dots `...` to display the Namespace menu. From the Namespace menu select `Persistent`.

### Garbage collection settings

The [Okteto Garbage Collector](self-hosted/helm-configuration.mdx#gc) service can automatically scale down [idle applications](admin/cleanup.mdx#what-counts-as-activity) and delete unused Namespaces for optimal cost savings and resource management.

Please see our [resource cleanup guide](admin/cleanup.mdx) to see the management options available and how to configure them.

### Configure Namespace Quotas

Administrators can manage resource allocation by setting [quotas](self-hosted/helm-configuration.mdx#quotas) for each Namespace.
These quotas limit the maximum resources that can be used per Namespace.

When enabled, the Okteto dashboard displays the following quotas on the Namespace view:

<p align="center">
  <Image
    src={require("@site/static/img/quotas.png").default}
    alt="Namespace quotas"
    width="500"
  />
</p>


- **CPU**: Configured with `quotas.requests.cpu`. Limits the total cpu resource requests per Namespace.
- **Memory**: Configured with `quotas.requests.memory`. Limits the total memory resource requests per Namespace.
- **Storage**: Configured with `quotas.requests.storage`. Limits the total volume storage per Namespace.
- **Pods**: Configured with `quotas.resources.maxPods`. Limits the total number of pods per Namespace.
