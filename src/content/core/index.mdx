Before you start building with Okteto, it helps to understand the core concepts that underpin the platform. This section covers the foundational building blocks — from how your environments are organized to how your code gets built, deployed, and accessed.

## Environment and access

### Namespaces
Organize your development environments into isolated workspaces. Each developer gets a personal namespace, and you can create shared namespaces for team collaboration.

[Learn more about Namespaces →](core/namespaces.mdx)

### Credentials
Access your Okteto environments through Kubernetes credentials, personal access tokens, or environment variables.

- [Kubernetes credentials](core/credentials/kubernetes-credentials.mdx) — connect `kubectl` and other tools to your Okteto namespace
- [Personal access tokens](core/credentials/personal-access-tokens.mdx) — authenticate CLI and API access
- [Environment variables](core/credentials/environment-variables.mdx) — manage secrets and configuration

### User roles and permissions
Okteto uses role-based access control (RBAC) with two primary roles: Admins and Developers. Understand what each role can do across the platform.

[Learn more about roles and permissions →](core/user-roles-and-permissions.mdx)

## Networking

### Endpoints
Okteto automatically generates HTTPS endpoints for your deployed services, with SSL certificates managed for you.

- [Automatic SSL](core/endpoints/automatic-ssl.mdx) — auto-generated HTTPS endpoints for your services
- [Private endpoints](core/endpoints/private-endpoints.mdx) — restrict access to internal services

### Divert
Route traffic intelligently across microservice environments. Deploy only the services you're modifying and connect to shared versions of everything else.

[Learn more about Divert →](core/divert.mdx)

## Build and configuration

### Okteto Manifest
The `okteto.yaml` file is the central configuration for building, deploying, testing, and developing your application in Okteto. It defines everything from build targets to dev container settings.

[Learn more about the Okteto Manifest →](core/okteto-manifest.mdx)

### Okteto Variables
Save application configuration values in Okteto and inject them automatically at deployment time. Manage variables at the namespace, user, or admin level.

[Learn more about Okteto Variables →](core/okteto-variables.mdx)

### Build service
Build container images remotely using BuildKit in your cluster. Images are automatically pushed to the Okteto Registry.

[Learn more about the Build Service →](core/build-service.mdx)

### Container registry
Each Okteto namespace has its own space in the built-in container registry to store and pull images.

[Learn more about the Okteto Registry →](core/container-registry.mdx)

### Remote execution
Run your deploy, test, and destroy commands in the cluster instead of on your local machine. This ensures consistent, reproducible operations.

[Learn more about Remote Execution →](core/remote-execution.mdx)

## Data and observability

### Insights dashboards
Monitor build times, deploy frequency, resource usage, and user activity across your Okteto cluster.

[Learn more about Okteto Insights →](core/okteto-insights-dashboards.mdx)

### Volume snapshots
Initialize persistent volumes with data from a previous snapshot. Useful for seeding development databases with realistic data.

[Learn more about Volume Snapshots →](core/use-volume-snapshots.mdx)
