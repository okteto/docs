AI agents can work with Okteto environments the same way human developers do — using the Okteto CLI, reading `okteto.yaml` for service discovery, and getting feedback from live environments to iterate on changes. No special infrastructure or per-repo configuration files are needed.

This section covers how to set up and use AI agents (such as [Claude Code](https://claude.ai/code)) with your Okteto environments, whether you're working alongside an agent in your IDE or letting an agent handle a task end-to-end.

:::note
This is different from [Okteto AI](okteto-ai/index.mdx), which is a standalone product for launching and managing agents from the Okteto Dashboard. Agentic workflows are about bringing your own agent tooling to work with Okteto environments directly.
:::

## Why Okteto for agents

**Isolated environments on demand.** Each agent gets its own Okteto namespace with dedicated compute, storage, and network. Agents can't interfere with each other or with production.

**`okteto.yaml` as the single source of truth.** Agents read the manifest to discover which services exist, how they're built, how they're deployed, and what tests are available. No hardcoded service lists or extra config files to maintain.

**Same tooling as humans.** Agents use the same CLI commands developers use — `okteto deploy`, `okteto build`, `okteto test`, `okteto logs`. If it works for a developer, it works for an agent.

**Live feedback loops.** Agents can deploy changes, hit live endpoints, read logs, and run tests against real infrastructure — not mocks or simulations.

**Preview environments.** Every PR can get its own running environment that stakeholders can access without local setup.

## Two operating modes

### Collaborative mode

A developer works alongside an agent in their IDE or terminal. The developer manages the development environment with `okteto up`, and the agent runs commands inside it using `okteto exec`.

Best for: pair programming with an agent, interactive debugging, feature development where you want to stay in the loop.

See [Collaborative Workflows](agentic/collaborative-workflows.mdx) for details.

### Autonomous mode

An agent handles the full lifecycle — from reading a ticket to opening a pull request — without human intervention. The agent uses `okteto deploy`, `okteto build`, and `okteto test` to manage the environment directly.

Best for: ticket-to-PR automation, CI-triggered agent pipelines, batch tasks like writing tests or fixing lint errors across a codebase.

See [Autonomous Workflows](agentic/autonomous-workflows.mdx) for details.

## The `okteto up` rule

`okteto up` starts an interactive development session with file sync and a live terminal. It requires human interaction and will hang indefinitely if run by an agent.

**Agents must never run `okteto up`.**

- In collaborative mode, the **human** runs `okteto up`. The agent uses `okteto exec` to run commands in the active dev container.
- In autonomous mode, the agent uses `okteto deploy` + `okteto build` to manage services directly. `okteto up` is not part of the workflow.
