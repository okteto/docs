This tutorial walks you through running unit tests and end-to-end tests in [Okteto Test Containers](/docs/testing/getting-started-test). You will deploy the Movies sample app, configure test containers for the frontend, API, and end-to-end layers, and execute each test suite against a live Okteto environment.

## What you will learn

- How to configure test containers in `okteto.yaml`
- How to run unit tests for individual services
- How to run end-to-end integration tests with Playwright
- How to use caches and artifacts to speed up test execution

## Prerequisites

Before you start, make sure you have:

- Access to an Okteto instance. Otherwise, follow the [installation guide](/docs/get-started/install/index)
- The [Okteto CLI configured](/docs/get-started/install-okteto-cli) with your Okteto instance
- Completed the [deploy your first app guide](/docs/get-started/deploy-your-app/index)

## Step 1: Clone and deploy the Movies app

Get a local copy of the Movies sample app:

```bash
git clone https://github.com/okteto/getting-started
cd getting-started
```

Switch to the `onboarded` branch, which includes pre-configured test containers:

```bash
git checkout onboarded
```

Deploy the application:

```bash
okteto deploy
```

## Step 2: Understand the test configuration

Open `okteto.yaml` at the root of the repository. The `test` section defines three test containers:

```yaml title="okteto.yaml"
test:
  frontend:
    image: node:20
    context: frontend
    caches:
      - yarn/.cache
      - node_modules
    commands:
      - yarn install
      - yarn test
  api:
    image: node:20
    context: api
    caches:
      - yarn/.cache
      - node_modules
    commands:
      - yarn install
      - yarn test
  e2e:
    image: ghcr.io/okteto/playwright:chromium
    context: e2e
    caches:
      - yarn/.cache
      - node_modules
    commands:
      - bash run_tests.sh
    artifacts:
      - test-results
      - playwright-report
```

Each test container has these fields:

| Field | Purpose |
|-------|---------|
| `image` | The runtime image for the test container |
| `context` | The folder synchronized between your local machine and the test container |
| `caches` | Folders persisted between runs to speed up execution |
| `commands` | The commands that run your tests |
| `artifacts` | Files exported to your local filesystem after execution (e2e only) |

The `frontend` folder also contains an `.oktetoignore` file that prevents synchronizing binaries, build artifacts, and the `node_modules` folder to the test container.

The e2e tests use the `OKTETO_NAMESPACE` and `OKTETO_DOMAIN` environment variables to construct the base URL. You can see how this works in the [Playwright config](https://github.com/okteto/getting-started/blob/main/e2e/playwright.config.js#L3).

Because the `artifacts` field is configured, the `test-results` and `playwright-report` folders are downloaded to your local machine after execution. Open the Playwright report to inspect test details.

:::tip
To access private services like MongoDB from your integration tests, use the hostname `mongodb.${OKTETO_NAMESPACE}`. See how this works in the [data helper](https://github.com/okteto/getting-started/blob/main/e2e/data/getStoredData.js#L5).
:::

## Step 6: Add a new test container

Create a new test container for a custom validation. Add the following to the `test` section of your `okteto.yaml`:

```yaml title="okteto.yaml"
test:
  smoke:
    image: curlimages/curl:latest
    commands:
      - curl -sf https://movies-${OKTETO_NAMESPACE}.${OKTETO_DOMAIN} > /dev/null
```

Run the smoke test:

```bash
okteto test smoke
```

This verifies that your Movies app endpoint is reachable. You can extend this pattern to add health checks, API contract tests, or any validation that runs against your live environment.

## Summary

You configured and ran three types of tests in Okteto Test Containers:

- **Unit tests** for the frontend and API services, each running in isolated containers with cached dependencies
- **End-to-end tests** using Playwright against a live deployment, with test artifacts exported locally
- **A custom smoke test** that validates endpoint availability

Test Containers run the same way in your development workflow and in [Okteto Preview Environments](/docs/previews/index), giving you consistent test results across your inner and outer development loops.
